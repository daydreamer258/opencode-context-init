# Project Memory Assistant（记忆助手 v2）

**对应原则：9（默认不主动写入；可建议但不擅自改）。**

监听用户陈述，在恰当时机**建议**跑 `/remember`——**绝不**自己动手追加。

---

## 启用前置条件

**两个条件都满足**才启用：

1. 项目根存在 `AGENTS.md`
2. 用户的最新发言是"陈述项目专属信息"，而非提问/调试/指令

无 `AGENTS.md`：完全静默——让 `project-context-bootstrap` 哨兵去提示初始化。

---

## 触发判断

### ✅ 应该建议：用户在陈述项目级事实

明确触发：
- "我们的约定是 ..."
- "我们项目里 ..."
- "我们团队都用 ..."
- "记住，这里 ..."
- "以后都按 ... 来"
- "这里要注意 ..."
- "FYI，... " / "By the way, ..."
- "顺便说，..." / "对了，..." 后跟一个事实陈述
- "其实这里有个坑：..."
- "我们这边的规则是 ..."

特征：
- 主语是"我们/项目/团队"或暗含
- 内容是**断言**，不是问题
- 内容涉及业务/约定/陷阱/边界/决策——**模型推断不出的事**（原则 1）

### ❌ 不应该触发

- **提问**："这里怎么改？" / "为什么用 X？"
- **指令**："帮我加 X 功能" / "改这段代码"
- **debug 讨论**："报错是因为 ..."（除非用户明确说"以后注意"）
- **泛泛感想**："代码有点乱" / "这库不错"
- **一次性临时决策**："这次先这样" / "暂时跳过"
- **明显可推导的事实**："项目用 Spring"（违反原则 1，不该记）
- **已记录的事**：如果能判断已在 AGENTS.md / docs/agent/ 中，不要重复

**判断不确定时，倾向不触发**——误打扰代价大于漏记。

---

## 触发后的行为

### 唯一动作：建议跑 `/remember`

用一两句话提出建议，**不解释、不动手**：

> 听起来这是一条值得记忆的项目级事实。建议你跑：
> ```
> /remember <你刚才说的内容>
> ```
> （或回复 `n` 跳过）

**就这样。** 不要：
- ❌ 询问"我帮你记到 X 文件好吗"
- ❌ 自己开始查重、格式化、写盘
- ❌ 长篇解释为什么应该记

如果用户回 `n`：立即停止，继续原话题。

---

## 节流

- 同一会话同一条建议**只建议一次**
- 同一会话总触发**不超过 3 次**
- 用户回 `n` 之后，本次会话**不再**对类似内容建议
- 不要在用户连续陈述时打断——等用户说完再统一建议

---

## 边界与禁忌（与 v1 的关键差异）

- ❌ **绝不自己写盘**——原则 9：默认不主动改写
- ❌ **绝不询问"我帮你 ..."**——保持纯建议姿态
- ❌ **绝不触发后再追问** "要不要改其他地方"
- ❌ 不要在无 AGENTS.md 项目中触发
- ❌ 不要在提问/指令/debug 中打断
- ❌ 不要替用户判断"重要不重要"（除非内容显然可推导）
- ❌ 不要在用户拒绝后追问

---

## 与 failure-loop 的分工

| Skill | 时机 | 来源 |
|---|---|---|
| `project-memory-assistant`（本 Skill） | 对话**进行中** | 用户口中说出的项目级事实 |
| `failure-loop` | 任务**结束时** | Agent 自己学到的教训 |

互补：本 Skill 抓"用户说的"，failure-loop 抓"Agent 学的"。**两者都只建议，从不写盘**。
