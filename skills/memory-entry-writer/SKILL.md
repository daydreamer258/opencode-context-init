# Memory Entry Writer（写入规范）

被 `/remember`、`/update-memory`、`/init-project`（Phase 3）共享的"如何写一条记忆条目"规范。

**对应原则：12（原子化）、17（写入有模板）。**

---

## 条目格式

每条记忆是 markdown 二级或三级标题下的一个块，格式：

```markdown
### <ID>: <标题>  [<类型>]
- 关联代码: <路径>（带行号最佳）
- 新增: YYYY-MM-DD
- 最后验证: YYYY-MM-DD
- 标签: <逗号分隔>

**事实/决策**: <一句话>

**为什么**: <原因 / 背景 / 权衡>

**反例**: <看似合理但行不通的方案>（可选）
```

### 字段规范

| 字段 | 必填 | 说明 |
|---|---|---|
| ID | ✓ | `G-NNN`（全局，在 `_global.md`）或 `M-NNN`（模块，在 `<module>.md`），文件内递增编号 |
| 标题 | ✓ | 8–15 字，概括事实，不重复 |
| 类型 | ✓ | `decision` / `constraint` / `gotcha` / `sop` 四选一 |
| 关联代码 | ✓ | 至少一个路径；不知道时填"全局"或"待补充" |
| 新增 | ✓ | 首次写入日期，**永不修改** |
| 最后验证 | ✓ | 用户明示有效的日期；`/update-memory` 时自动刷新 |
| 标签 | ✓ | 3–6 个关键词，用逗号分隔，全小写英文或中文短词 |
| 事实/决策 | ✓ | 一句话核心内容 |
| 为什么 | 强烈建议 | `decision` 和 `gotcha` 必须；`constraint` 和 `sop` 强烈建议 |
| 反例 | 可选 | 用于 `decision` 和 `gotcha` |

### 4 种类型语义

| 类型 | 含义 | 示例 |
|---|---|---|
| `decision` | 团队做过的、对未来有约束的选择 | "选 HikariCP 作连接池"、"用领域事件而非直接 RPC" |
| `constraint` | 不能这样做的规则（含正面禁令） | "余额必须用 Decimal"、"不能直接 setStatus" |
| `gotcha` | 痛苦换来的知识——踩过的坑 | "OAuth refresh 并发会死锁"、"V*.sql 已上线版本不能改" |
| `sop` | 做某件事的标准步骤 | "加新事件类型的 5 步"、"发版前的检查清单" |

## ID 生成

文件级递增。流程：

1. 读目标文件，找出所有现有条目 ID（regex 匹配 `^### (G|M)-\d+:`）
2. 取最大编号 + 1，前缀按文件类型决定
3. 编号宽度 3 位（`G-001`、`G-002`...），不要 `G-1`

**注意**：删除条目时**不要**重排其他 ID（保持稳定）。新写入永远用"现有最大 +1"，不复用已删除的 ID。

---

## 章节归属

按类型映射到目标文件章节：

| 类型 | `_global.md` 章节 | `<module>.md` 章节 |
|---|---|---|
| `decision` | "历史重大决策与事故" 或 "全局约定" | "模块设计与决策" |
| `constraint` | "全局约定" / "数据约定" / "对外 API 约定" | "约束与边界" |
| `gotcha` | "历史重大决策与事故" | "陷阱与历史坑" |
| `sop` | "SOP（操作流程）" | "本模块 SOP" |

如果目标文件**没有**对应章节（用户在 `/init-project` 时该章节空白被删了），先加章节标题再追加条目。

## 写入前的检查（dedupe + filter）

被 `/remember` 调用时，**先**做：

### 1. 内容过滤（原则 1、3、19）

拒绝下列内容直接写入：

- **可推导**：可以从 `README` / 配置文件 / 目录结构 / 代码本身直接读出来的事实
- **秘密/快变**：密钥、token、密码、连接字符串、当前版本号、临时联系人

判断不确定时**询问用户**，不要默默 ban。

### 2. dedupe 搜索（原则 10）

跨 `docs/agent/*.md` 搜索语义相近的条目。比较维度：
- 标题相似度
- 事实文本相似度
- 标签重叠度

找到则要求用户决策：跳过 / 新增独立条目 / 跳到 `/update-memory`。

### 3. 冲突检测（原则 10）

如新条目与已有条目**事实矛盾**（一个说"必须 X"、另一个说"必须 Y"）：
- 提示用户：是 supersede（旧的过期，先 `/forget` 再写）、还是 coexist（场景不同，加 `supersedes` 或 `coexists-with` 标签注明）

---

## 写入流程

1. 用户/命令确定目标文件 + 章节
2. 过滤 + dedupe + 冲突检测通过
3. 用上述格式生成条目
4. 给用户预览
5. 确认后**追加**到目标文件对应章节末尾
6. 更新该文件 frontmatter 的"最后更新"为今日日期
7. **不**自动 git

---

## 反模式（不要写成这样）

❌ **长段落记忆**：
```markdown
我们这个项目用了一个比较复杂的 OAuth 流程，主要是因为之前有用户反馈
登录慢，后来排查发现是 token refresh 的并发问题，所以现在改成了 ...
```

✅ 改成原子条目：
```markdown
### M-003: OAuth refresh 并发死锁  [gotcha]
- 关联代码: src/auth/oauth.ts:42-58
- 新增: 2026-05-26
- 最后验证: 2026-05-26
- 标签: oauth, race-condition, auth

**事实**: 同一 client_id 并发 token refresh 会触发 provider 端死锁
**为什么**: provider 对 client_id 加锁，并发请求互相等待
**反例**: 用 Promise.all 并发刷新——会死锁
```

❌ **写代码里能看到的事**：
```markdown
### G-001: 用 Spring Boot 框架
```

✅ 该删（违反原则 1）。
