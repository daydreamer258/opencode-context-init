# Failure Loop（失败回路）

**对应原则：18（失败回路——任务结束时自检）。**

> 当 Agent 在任务中遇到"如果当初有记忆就不会犯的错误"时，应在任务结束时主动提示用户考虑添加记忆。这是记忆资产自然增长的主要来源。

---

## 启用前置条件

**同时满足**才启用：

1. 项目根存在 `AGENTS.md`
2. 当前任务**已经结束**（用户的请求已完成或显式停止）

**任务进行中绝不触发**——避免打断当前工作。

---

## 触发时机的判断

"任务结束"的信号：

- 用户说"好的，搞定了" / "谢谢" / "完成了"
- 用户切换话题到无关内容
- 用户说"先这样"、"够了"
- Agent 给出完整解决方案并用户表示满意
- 一段对话明显告一段落

**不要**在以下情况触发：
- 用户还在追问当前问题
- 还在 debug / 调整中
- 用户在抱怨/烦躁——这时候提建议会火上浇油

---

## 自检流程

任务结束时，**默默**回顾本次会话，检查：

### 1. 是否踩了"早知道就好"的坑

特征：
- 中途出错过，然后从用户那里学到了一个项目特有规则
- 最初按通用做法写代码，后被用户纠正到特定约定
- 改了某文件才发现"这文件不该改"
- 看了 N 个文件才搞清楚某个业务逻辑（原则 2：探索成本阈值）

如果**任一**命中：候选记忆 1 条。

### 2. 是否发现了未文档化的约定

特征：
- 用户在对话中陈述了"我们项目里 X"、"我们约定 Y"
- 但 `AGENTS.md` 和 `docs/agent/` 都没记
- 且这条信息**不是从代码能直接读的**

如果命中：候选记忆 1 条。

### 3. 是否解决了一个有"为什么"的问题

特征：
- 修了一个 bug，并且知道根因
- 改了一个设计选择，并且知道权衡

这种 `decision` 或 `gotcha` 类型的知识值得记。

---

## 触发后的行为

如果有候选记忆，**简短**告知用户：

```
💡 任务结束自检：

本次我注意到 N 条可能值得记忆的事：

1. [gotcha] 我最初想直接改 src/legacy/foo.ts，你提醒我这是只读区
   → 这个边界你想记到 AGENTS.md § 2 禁区吗？

2. [decision] 我们把 X 改成了 Y，因为 Z
   → 这个决策你想记到 docs/agent/<module>.md 吗？

你可以：
  - 跑 /remember <内容> 单独记某一条
  - 全部跳过

（这是建议——不要打断你的节奏）
```

如果**没有**候选记忆：完全静默，**不要**为了"显得有用"硬找内容。

---

## 节流与禁忌

- ❌ 不要在任务进行中触发——只在结束时
- ❌ 不要列超过 3 条候选——保持简短
- ❌ 不要自动写盘——只建议，由用户跑 `/remember`
- ❌ 不要在用户说"先这样"或"算了"后追问
- ❌ 不要把通用建议（如"加点测试"）当候选记忆
- ❌ 不要列已经在 `AGENTS.md` 或 `docs/agent/` 里的内容
- ❌ 一次会话最多触发一次

---

## 与 project-memory-assistant 的分工

| Skill | 触发时机 | 来源 |
|---|---|---|
| `project-memory-assistant` | 对话**进行中**，用户陈述项目级事实时 | 用户主动陈述 |
| `failure-loop`（本 Skill） | 任务**结束时**，Agent 自检 | Agent 自己学到的教训 |

两者互补：前者捕获用户**说**的，后者捕获 Agent **学**的。
