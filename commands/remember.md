# /remember — 追加项目知识

用户用这个命令显式告诉你"这条信息要记住"。你的任务：判断该写到哪个文件、格式化好、确认后追加。

**核心原则：**
- 全程中文交互
- **追加**，不要覆盖已有内容
- 写入前必须给用户预览并确认
- 不要重复——如果信息已存在，告知用户而不是再写一遍

---

## Phase 0：前置检查

1. 检查项目根目录是否存在 `AGENTS.md`。
   - **不存在**：告知用户先运行 `/init-project` 初始化项目知识库，然后退出。
   - **存在**：继续。

2. 获取"要记住的事"：
   - 如果用户在 `/remember` 后跟了内容，直接用
   - 如果只输入了 `/remember`，反问用户："你想让我记住什么？请告诉我具体内容。"

---

## Phase 1：分类（决定写到哪个文件）

按以下顺序判断目标文件。**先匹配标准 5 个**，匹配不上再看可选文档，最后才考虑新建。

### 1.1 标准映射表（优先）

| 信息性质 | 目标文件 |
|---|---|
| 业务定义、术语解释、产品意图、上下游系统 | `docs/agent/overview.md` |
| 构建命令、运行步骤、代码生成流程、环境依赖、排错步骤 | `docs/agent/build-and-run.md` |
| 代码风格、命名规则、包结构、注入方式、错误处理、测试约定 | `docs/agent/conventions.md` |
| 陷阱、坑点、历史 bug、"看起来能改但改了出问题"的地方 | `docs/agent/gotchas.md` |
| 不能改的目录、敏感文件、只读区、需要征求确认的区域 | `docs/agent/boundaries.md` |
| 给 Agent 的核心行为规则（罕见，只在用户明确说"这是元规则"时） | `AGENTS.md`（根） |

### 1.2 扫描可选文档

读取 `docs/agent/` 下所有 `.md` 文件，**排除**标准 5 个。对剩余的可选文档：

- 看每个文件**第一行的标题**和**前几行的说明**，理解它涵盖什么主题
- 如果用户给的信息明显属于某个可选文档的主题（如涉及"如何验证" → `verification.md`，涉及"部署" → `deployment.md`），把该文档加入候选列表

### 1.3 决定目标

- **唯一匹配**：直接用，进入 Phase 2
- **多个候选**（标准映射 + 可选文档都可能）：列出候选让用户选，参见 1.5
- **零候选**：进入 1.4 提议新建

### 1.4 提议新建文档

如果信息**明显不属于现有任何文档**（例如用户说的是"合规要求"，但项目没有 compliance 相关文档），询问：

> 这条信息我没找到合适的现有文档归档。建议：
>   1. 强行塞入 `docs/agent/<我推荐的最接近的标准文档>`
>   2. 新建一个文档 `docs/agent/<我建议的名字>.md` 专门记这类信息（你也可以改名字）
>   3. 取消，这次不记
> 你选哪个？

如果用户选 2：
- 让用户确认或修改文件名
- 用 `templates/optional/_custom.md.tmpl` 作为骨架创建文件（含基础结构）
- 然后追加用户给的信息

### 1.5 模糊归类时

如果一条信息可以放多个文件（例如"我们用 HikariCP 连接池"既可能是约定也可能是构建相关），列出 2–3 个候选让用户选：

> 这条信息可以放：
>   1. `docs/agent/conventions.md`（视为团队约定）
>   2. `docs/agent/build-and-run.md`（视为运行配置）
> 你选哪个？（1/2）

---

## Phase 2：查重

读取目标文件，检查是否已包含相同或相近的信息。

- **完全重复**：告知用户"已存在于 `<文件>` 第 N 行附近，无需重复添加"，退出
- **相近但不完全相同**：展示已有内容，问用户"已有这条相近内容：`<引用>`。是要**追加新条目**，还是**修改已有条目**？"
- **不存在**：继续 Phase 3

---

## Phase 3：格式化

不同文件有不同结构。**保持与目标文件现有风格一致**：

### `overview.md`
- 术语：加到"关键术语"表格
- 业务定义/流程：作为新段落追加到相关章节末尾

### `build-and-run.md`
- 命令：加到"常用快捷命令"表格
- 代码生成：作为"代码生成流程"章节的新条目（4 要素：来源/触发/输出/AI 规则）
- 排错：加到"排错"列表

### `conventions.md`
- 命名/风格：加到对应表格
- 规则：作为对应章节的新 bullet

### `gotchas.md`
- 新陷阱：作为新的"陷阱 N"小节，遵循 4 要素：现象、原因、正确做法、相关文件
- 历史教训：加到"历史 Bug 教训"列表

### `boundaries.md`
- 禁区：加到"绝对禁区"表格
- 敏感文件：加到"敏感文件"表格
- 只读区：加到"只读区"列表

### `AGENTS.md`（仅在元规则时）
- 元规则：加到"给 Agent 的元规则"编号列表末尾

### 可选文档（如 `verification.md` / `deployment.md` 等）
- 读目标文件已有的小节结构，把新内容插入到**主题最相关**的小节
- 找不到完全对应的小节就追加到文件末尾，必要时加新小节标题

### 新建的自定义文档
- 文件创建后已有 `_custom.md.tmpl` 的骨架（标题、用途、小节、AI 约束）
- 把用户给的信息填入相应的"小节"占位符
- 如果用户没说清楚要哪些小节，先放在"内容"或"备注"段，等后续 `/remember` 时再组织

---

## Phase 4：预览 + 确认

向用户展示：

```
我准备追加到：docs/agent/<文件名>
位置：<章节>
内容预览：

<格式化后的新内容>

确认追加？（y / 修改 / 取消）
```

- `y`：写入，进入 Phase 5
- `修改`：让用户描述修改点，重新格式化，回到 Phase 4
- `取消`：放弃，不写入

---

## Phase 5：写入

1. **追加**到目标文件的合适位置（不要覆盖整个文件）
2. 如果目标文件不存在（用户可能删了某个 docs/agent/*.md），创建它并加上文件级标题
3. 输出确认：

```
✓ 已追加到 docs/agent/<文件名>
```

4. 结束命令。**不要**自动 git add / commit。

---

## 边界与禁忌

- ❌ 不要把多条信息塞到一次 `/remember`——如果用户给的内容包含多个独立事实，提示用户分多次执行
- ❌ 不要修改用户没要求改的部分
- ❌ 不要在用户拒绝/取消后偷偷保留草稿
- ❌ 不要因为信息"听起来不像项目级知识"就拒绝——用户说要记，就帮记。Agent 不是把关人，是助手。
- ❌ 不要把同一条内容写到多个文件
