# /update-memory — 更新已有记忆条目

支持原则 10（写入前 dedupe 后选择更新而非新增）和原则 11（条目应有时间戳，更新时刷新"最后验证"）。

---

## Phase 0：前置检查

1. 检查 `AGENTS.md` 是否存在。**不存在**：告知"还没有记忆库"，退出。
2. 获取目标 ID：
   - 用户给了 ID：直接用
   - 没给：反问"你要更新哪条？给 ID 或关键词"

---

## Phase 1：定位条目

按 ID 跨 `docs/agent/*.md` 和 `AGENTS.md` 搜索。

- 找到：进入 Phase 2
- 找不到：告知"未找到 ID `<X>`"，可建议改用 `/forget <关键词>` 搜索

---

## Phase 2：展示当前内容 + 询问改什么

```
当前条目：

  文件：docs/agent/<文件>
  <ID>: <标题>  [<类型>]
  - 关联代码: <路径>
  - 新增: <旧日期>
  - 最后验证: <旧日期>
  - 标签: <旧标签>

  事实: ...
  为什么: ...
  反例: ...（如有）

你要改哪部分？（描述要怎么改，例如"事实改成 ..."、"加一个标签 race-condition"、"代码路径改成 ..."）
```

等用户描述修改点。

---

## Phase 3：应用修改

按用户描述生成新版本。**注意**：

- **保留** 条目 ID（不要换 ID）
- **保留** "新增"日期（这是首次写入的时间戳，永不改）
- **自动更新** "最后验证" 字段为今日日期
- 其他字段按用户描述修改

---

## Phase 4：预览（双栏对比）

展示前后对比：

```
变更预览：

字段          | 旧                 | 新
--------------|--------------------|--------------------
标题          | <旧>               | <新>
关联代码      | <旧>               | <新>
最后验证      | <旧日期>           | <今日>（自动）
标签          | <旧>               | <新>
事实          | <旧>               | <新>
为什么        | <旧>               | <新>

确认（y / 继续修改 / 取消）？
```

只显示**有变化**的字段，其他字段省略。

- `y`：进入 Phase 5
- `继续修改`：回 Phase 2 让用户再描述
- `取消`：放弃

---

## Phase 5：写入

1. 在目标文件中**原地替换**该条目（不要追加新条目，不要换位置）
2. 更新目标文件 frontmatter 的"最后更新"为今日日期
3. 输出确认：

```
✓ 已更新 docs/agent/<文件> 条目 <ID>
  最后验证 → <今日日期>
```

4. **不要**自动 git。

---

## 边界与禁忌

- ❌ 不要改"新增"日期（这是历史，永不改）
- ❌ 不要把更新做成"删旧 + 新增"——这样 ID 就变了
- ❌ 不要在用户取消时偷偷保留草稿
- ❌ 不要一次更新多条
- ❌ 不要自动 git
